  1. Query 1

    Make a request to get attributes from the specified tables, applying filters according to the specified conditions:

    • Tables: Н_ОЦЕНКИ, Н_ВЕДОМОСТИ. Display
    • Attributes: Н_ОЦЕНКИ.ПРИМЕЧАНИЕ, Н_ВЕДОМОСТИ.ЧЛВК_ИД.
    • Filters (AND):
      • Н_ОЦЕНКИ.КОД < осв.
      • Н_ВЕДОМОСТИ.ДАТА > 2010-06-18.
    • Type of join: LEFT JOIN.

    SQL Query:

    SELECT Н_ОЦЕНКИ.ПРИМЕЧАНИЕ, Н_ВЕДОМОСТИ.ЧЛВК_ИД
FROM
    Н_ОЦЕНКИ LEFT JOIN Н_ВЕДОМОСТИ ON Н_ОЦЕНКИ.КОД = Н_ВЕДОМОСТИ.ОЦЕНКА
WHERE
    Н_ОЦЕНКИ.КОД < 'осв'
    AND Н_ВЕДОМОСТИ.ДАТА > '2010-06-18';

    a. Indexes

    CREATE INDEX ix_date ON Н_ВЕДОМОСТИ USING btree(ДАТА);

    The reason for using the Н_ВЕДОМОСТИ.ДАТА b-tree index is that the query uses the “>” operator and the number of Н_ВЕДОМОСТИ records is large → this will help make the query faster

    Do not use a b-tree index for Н_ОЦЕНКИ.КОД as the number of records in the Н_ОЦЕНКИ table is small, using an index will only increase the overhead cost without increasing efficiency

    b. Plan

    Simple plan

    Because in the filter two predicates are used, each from a different table, after performing the JOIN operation, the result will be equivalent to an inner join.

    To still use LEFT JOIN while maintaining the result, we can perform one predicate before JOIN and one after JOIN

Noting that the sooner we use projection, the better performance we receive, we can rewrite the plan by using projection from the beginning

Because the final result will be equivalent to INNER JOIN, we can swap the positions of relations when JOIN to optimize performance, the outer relation will be the table with more rows)

  1. Query 2

    Make a request to get attributes from the specified tables, applying filters according to the specified conditions:

    • Tables: Н_ЛЮДИ, Н_ВЕДОМОСТИ, Н_СЕССИЯ.
    • Display attributes: Н_ЛЮДИ.ФАМИЛИЯ, Н_ВЕДОМОСТИ.ЧЛВК_ИД, Н_СЕССИЯ.УЧГОД.
    • Filters (AND):
      • Н_ЛЮДИ.ИМЯ > Николай.
      • Н_ВЕДОМОСТИ.ИД > 1250981.
      • Н_СЕССИЯ.ИД > 32199.
    • Type of join: INNER JOIN.

    SQL Query

    SELECT Н_ЛЮДИ.ФАМИЛИЯ, Н_ВЕДОМОСТИ.ЧЛВК_ИД, Н_СЕССИЯ.УЧГОД
FROM
    Н_ЛЮДИ
    INNER JOIN Н_ВЕДОМОСТИ ON Н_ВЕДОМОСТИ.ЧЛВК_ИД = Н_ЛЮДИ.ИД
    INNER JOIN Н_СЕССИЯ ON Н_СЕССИЯ.ЧЛВК_ИД = Н_ЛЮДИ.ИД
WHERE
    Н_ЛЮДИ.ИМЯ > 'Николай'
    AND Н_ВЕДОМОСТИ.ИД > 1250981
    AND Н_СЕССИЯ.ИД > 32199;

    a. Indexes

    CREATE INDEX ix_name ON Н_ЛЮДИ USING btree(ИМЯ);
CREATE INDEX ix_id ON Н_ВЕДОМОСТИ USING btree(ИД);
CREATE INDEX ix_id ON Н_СЕССИЯ USING btree(ИД);

    Use b-tree indexes for the columns of the above tables because the number of records in these tables is large and the query uses the “>” operator.

    b. Plan

    Simple plan

    As we are using INNER JOIN, swapping the order of JOINs does not change the final result.

    Similarly to task 1, to optimize performance when JOINing, the outer relations will be the tables with more records. Here, we have a one-to-many relationship between Н_ЛЮДИ and Н_ВЕДОМОСТИ.

    Just like task 1, the sooner we perform projection and predicate, the better performance we receive.
